# 06 · ナレッジ学習機構（この製品の心臓部）

> Grow の差別化の核。「タスクの履歴が資産になり、AIが使うほど自分の働き方にフィットしていく」。依頼者の要件を、実装可能な設計に落とし込む。**主戦場は「AIの実作業アウトプットの質」**（壁打ちの質向上は副次。理由は依頼者曰く、壁打ちは各人のパターンが多様で汎化しにくく、ゼロから始まる業務では適用先が無いこともあるため）。

---

## 6.1 3つのレイヤー（混同しないこと）

やりたいことは3つの別物の連携。分けて設計する。

1. **エピソード記憶（Episodic）= 生の履歴。** 各タスクの `comments` / `chat_messages` / ステータス遷移 / 差し戻し。**看板が自然に貯める。** 追加実装はほぼ不要（すでにDBにある）。
2. **意味記憶（Semantic）= 蒸留したルール。** `rules` テーブル。①から抽出した再利用可能な「判断軸・前提・進め方」。これが依頼者の言う「別のストア」。
3. **検索・適用（Retrieval / Apply）= 前提注入。** 次にAIが作業する時、②から関連ルールを引いてプロンプトに混ぜる（§07）。

**ループ:** ①が貯まる → ②へ蒸留 → ③でAIの作業に効く → 新たな①が貯まる …。価値は②の蒸留精度と③の適用精度で決まる。①は放っておいても貯まる。

---

## 6.2 ルール（意味記憶）のスキーマと意味

§02 の `Rule` を参照。要点:
- `text`: AIに注入する自然文の指示（例「レポートは結論→根拠の順で書き、冒頭に3行サマリーを置く」）。**命令形で、検証可能な粒度**が良い。
- `scope`: `personal`（個人の判断軸）/ `team`（チームの形式知）。**scope が後のチーム展開の鍵**。個人で貯めて、良いものをチームに昇格。
- `tags`: **空＝全体ルール（常に適用）**、非空＝そのラベルを持つタスクにのみ適用。retrieval のキー。
- `source` / `source_task_id`: どこから学んだか。信頼性の説明可能性（なぜAIがこう振る舞うか）を担保。
- `confidence`: high/med/low。適用実績・矛盾・人の評価で変動しうる（§6.6）。
- `applied` / `last_applied_at`: 使われた頻度と鮮度。降格・棚卸しの判定材料。

---

## 6.3 適用（Retrieval）— 主戦場

AIが作業に入る**前**に、該当ルールを集めて system プロンプトに注入する（§07）。

**MVPの検索ロジック（タグ一致・決定的）:**
```
relevantRules(task) =
  rules.filter(r =>
     r.tags.length === 0                       // 全体ルールは常に適用
     || r.tags.some(t => task.labels.includes(t))  // タグがタスクのラベルと交差
  )
```
- personal / team 両方を対象にする（チーム化後は「そのタスクの owner の personal ＋ workspace の team」）。
- 適用したら各ルールの `applied++`、`last_applied_at` 更新、`rule_applications` に記録。
- UI（§03.3.2 (b)）で「AIが着手時に前提にするルール」として**必ず可視化**する。ブラックボックスにしない＝人が「なぜこう動くか」を理解し、要らないルールに気づける。

**将来（フェーズ2以降）: 意味検索。** タグ一致だけだと取りこぼす。`rules.embedding`（pgvector）にルール文の埋め込みを持ち、タスクのタイトル＋説明＋履歴の埋め込みと近傍検索。ハイブリッド（タグ一致 ∪ ベクタ近傍 top-k）にし、注入件数は上限（例 8件）＋ confidence 降順で足切り。

**注入の設計指針:**
- 全部を無条件で入れない。**上限8件・confidence 降順で足切り**（§00 #8 で確定）。トークンにも上限。confidence high を優先。
- 矛盾するルールが同時に引かれたら、新しい/高確度を優先し、矛盾は §6.6 で検出・解消。

## 6.4 蒸留（Distillation）— 履歴からルールを作る

**MVPは「手動蒸留」から始める。いきなり定期ジョブ全自動にしない。** 理由（依頼者と合意した設計思想）:
- 蒸留の質を評価する術が無い段階で自動化すると、**間違った学びが静かに蓄積してAIが変な癖を覚える**。
- まず人が蒸留ループに1回噛み、採用/却下の判断を貯めてから自動化する。

### フェーズ2a: 手動蒸留（MVPで作る）
- トリガー: タスクが `you_review`/`reviewing`/`done` に達したカードの「✧ 学ぶ」ボタン（§01.7, §03.3.2c）。
- 処理: AIがそのタスクの履歴（コメント・壁打ち・差し戻し）を読み、**ルール候補を1〜数件提案**（§07 の distill プロンプト）。各候補に scope/tags/confidence を付ける。
- 承認: 人が候補ごとに **採用/却下/（編集）**。採用のみ `rules` に永続化。
- **人の採用/却下 log 自体を保存**（`rule_feedback` 等）。これが後の自動化のお手本データになる。

### フェーズ2b: 半自動（フェーズ2後半）
- タスク完了時に候補を**自動生成して通知だけ**（バッジ/受信箱）。人はまとめてレビュー。承認は人。

### フェーズ3: 自動 ＋ 自己修正（チーム化以降）
- 定期ジョブ（CI/定期バッチ）で、直近完了タスク群をまとめて蒸留。
- **ただの追加ではなく**: 既存ルールとの**重複統合**、**矛盾検出**、**使われないルールの降格・アーカイブ**を必ず入れる（§6.6）。これが無いと自動化は必ず腐る。
- 高頻度・高確度で安定したものだけを自動採用、曖昧なものは人レビュー行きにする閾値運用。

> **重要:** 定期ジョブ/CIは「仕組み」から入る発想で、フェーズ3まで作らない。MVPは手動蒸留＋適用で「使うほど1つずつルールが増え、次から言わなくてよくなる」最小ループを1周させることが目的。

---

## 6.5 個人 → チーム 昇格（形式知化）
- 個人ルールの「チームへ昇格 ↑」で scope=`team` に（§01.8, §03.4）。
- 意味: あなたの暗黙知がチームの共有形式知になる。チーム化後は全メンバーの retrieval 対象に。
- フェーズ3の発展: チーム内で同じ個人ルールが複数人に出現したら「チーム昇格を提案」。逆にチームルールに個人が例外を持てる「上書き（override）」も将来検討。

---

## 6.6 確度ライフサイクル・矛盾検出・降格（腐らせない仕組み）
自動化フェーズで必須。MVPでは概念だけ持ち、簡易実装で可。
- **確度の上下:** 採用直後 med。適用が続き差し戻しが起きなければ昇格（→high）。適用したのに差し戻し/明示的否定が続けば降格（→low→アーカイブ提案）。
- **鮮度:** `last_applied_at` が古く `applied` が伸びないルールは棚卸し候補。
- **矛盾検出:** 新ルールが既存ルールと意味的に矛盾（例「敬体で」vs「常体で」）→ 埋め込み類似＋LLM判定で検出し、人に「置き換え/共存/破棄」を提示。
- **説明可能性:** どのルールがどのアウトプットに効いたかを `rule_applications` で追える。「このレポートはK-01,K-03に基づき生成」と示せる。

---

## 6.7 プライバシー・セキュリティ（チーム化で重要）
- personal ルールは owner 本人のタスクにのみ既定適用。無断で他人へ漏らさない（昇格は本人の明示操作）。
- team ルールでも、機微情報（個人の判断の癖・評価）を含めない蒸留プロンプト設計にする（§07）。
- AIの実行権限（Web/ファイル/外部API）とデータ境界を workspace 単位で制御。監査ログ（誰の/どのデータにAIが触れたか）を残す。
- 蒸留は生ログから「一般化された指示」だけを取り出す。固有名詞・秘密情報をルール文に焼き込まない（プロンプトで明示）。

---

## 6.8 受け入れ基準（この機構が「効いている」と言える条件）
1. タスクを完了→「✧ 学ぶ」→採用 で `rules` が1件増え、ナレッジ一覧に NEW で出る。
2. そのルールの tags に一致する新タスクで「AIにまかせる」を押すと、適用ルールに**そのルールが表示**され、AIが「〜を前提に着手します」と宣言する。
3. 同じ指示を繰り返さなくても、2回目以降のアウトプットに前回のルールが反映されている（実AI接続後に検証）。
4. 個人ルールを「チームへ昇格」すると scope=team になり、（チーム化後）他メンバーの該当タスクでも適用される。
