# 引き継ぎ書: Grow — 人とAIの協働タスク管理（看板ボード）

> このフォルダは、Claude Code（または実装担当者）が Grow をゼロから実装できるようにするための **完全な引き継ぎパッケージ** です。フロントエンドの見た目だけでなく、**プロダクトのコンセプト・協働フロー・バックエンド機構（ナレッジ学習）・データモデル・AI連携・MVPの進め方** まで、この一式だけで実装判断ができるように書いています。

---

## 0. 最初に読むべきこと（3分サマリー）

**Grow は「使うほど自分の働き方にフィットしていく」タスク管理ツール** です。見た目は看板ボード（Kanban）ですが、本質は次の2点にあります。

1. **人とAIが1枚のカード上でバトンを渡し合う。** タスク＝カード。人がコメントで依頼するとAI（または人）が着手し、AIは進捗を同じカードにコメントしていく。人はいつでも介入でき、AIは「完了、レビューはお任せします」と人にボールを返す。誰にボールがあるかが常に色で分かる。

2. **試行錯誤の履歴が資産になる（＝使うほど賢くなる）。** カードに貯まるやり取りの履歴から、AIが「働き方のルール（判断軸・前提知識）」を蒸留して別ストア（ナレッジ）に書き出す。次にAIが作業するとき、そのルールを **言われなくても前提として** 読み込んでから動く。結果、AIの実作業アウトプットの質が上がっていく。個人の暗黙知はチームの形式知へ「昇格」できる。

> **最重要の設計思想（依頼者の言葉）:** 主戦場は「AIの実作業アウトプットの質」。壁打ちの高度化は各人でパターンが多様すぎて汎化しにくいが、アウトプットは判断軸を前提化するほど確実に効く。だから「**AIが着手する前に、あなた／チームのルールを読んでから動く**」体験をプロダクトの中心に置く。

想定ユーザーはまず **個人**（依頼者自身）。個人ツールとして成立したら **チーム → 部署 → 全社** へ広げる。データモデルと権限は最初からこの拡張を見据えて設計する（が、認証・チーム機能の実装自体はMVPでは後回し。§ 08 参照）。

## 1. このバンドルのファイルについて（重要）

このフォルダの `.dc.html` ファイルは **HTMLで作られたデザインリファレンス（プロトタイプ）** です。**本番コードとしてそのままコピーするものではありません。** 意図した見た目・挙動・情報設計を示すための試作です。

実装タスクは、**これらのデザインを、対象コードベースの環境（React / Vue など）で、その確立されたパターン・ライブラリを使って作り直すこと** です。まだコードベースが無いため、本書 § 3.0 で推奨スタックを提示しています。最終的な技術選定は実装者の判断で構いませんが、**プロダクトのコンセプト・フロー・データモデル・ナレッジ機構（§ 01, 02, 06, 07）は仕様として厳守** してください（ここがこの製品の価値の源泉です）。

### 忠実度（Fidelity）
UIプロトタイプは **ハイファイ（hi-fi）** です。色・タイポグラフィ・余白・角丸・影・アニメーションは意図した最終値に近いものです。§ 03 / § 04 の値をそのまま使ってピクセル忠実に再現してください。ただし**アイコンはプレースホルダ**（`◈ ◎ ✧ ↑ ✕` などの文字）で描いています。実装ではアイコンライブラリ（例: Lucide, Phosphor）の適切なアイコンに置き換えてください（§ 03 に対応表）。

## 3.0 アーキテクチャ（GCP 前提で確定 — 詳細は §00）

> **本節の旧「Anthropic Claude 前提」アーキ図は §00 の GCP 構成で確定・上書きされた。** 本プロジェクトは GCP ハッカソン前提のため、ランタイムAIは Vertex AI 上の Gemini、ローカル開発は費用ゼロの Mock。判断軸は「コスト最小化 × 有名どころの順当なリソース」。完全な図と選定理由は **`00_decisions_and_platform.md` §0.2** を参照。

確定スタックの要点（詳細 §00）:

```text
Frontend  : React + TS + Vite / Zustand / @dnd-kit / TanStack Query / SSE
Backend   : FastAPI (Python) on Cloud Run（scale-to-zero、SPA静的配信も兼任＝1サービス）
Job Runner: Cloud Tasks → Cloud Run worker（Redis不要でコスト最小）
Data      : Cloud SQL for PostgreSQL（最小ティア）＋ 将来 pgvector
AI (B)    : Vertex AI Gemini（Function Calling / Web検索=Google Search グラウンディング）
補助      : Secret Manager / Artifact Registry / (将来)Cloud Storage / (P3)Firebase Auth
```

**なぜこの構成か:**
- **AIの実作業は非同期・長時間・失敗しうる** → API リクエスト内で完結させず、必ずジョブキュー（Cloud Tasks）に載せる。進捗は SSE でカードにpushし、失敗時はリトライ＋人へ通知（§05, §07）。
- **Redis を使わない**（Cloud Tasks + Cloud Run）ことで Memorystore 課金を避けコスト最小化。
- **ナレッジは「別ストア」** という依頼者の要件をそのまま `rules` テーブルとして分離。検索はMVPではタグ一致で十分、将来 pgvector で意味検索に拡張（§06）。
- **状態が正規化されている**（cards を id 辞書、lanes は cardIds 配列で持つ）ので Zustand/Redux と相性が良い。プロトタイプも同じ構造（§02, §05）。
- **ランタイムAIは Vertex AI Gemini。ローカルは Mock**（`AiProvider` 抽象で切替、§00 / §07）。追加費用は本番動作検証時のみ。
- **認証はMVPでは後回し可**（自分しか使わない）。ただしDBには最初から `workspace_id` / `owner_user_id` を持たせ、チーム化で作り直さない（§02, §08）。

## 5. 用語集

- **カード / タスク**: 作業の単位。コメント履歴を持つ。
- **サブタスク**: 分解で生まれた子カード（`parent_id` を持つ通常のタスク）。
- **ボール（の所在）**: いま次のアクションを取るべき主体（AI か 人）。カード左の色バーと上部カウンタで表現。
- **バトンを渡す / ハンドオフ**: 主体が AI↔人 で切り替わること（例: AIが「完了、レビューをどうぞ」＝人へハンドオフ）。
- **壁打ち（かべうち）**: 抽象的・大きいタスクを、AIとの対話で具体化・分解すること。
- **ブレイクダウン / 分解**: 壁打ちの結果を子タスク群に変換すること。
- **ナレッジ / ルール**: 履歴から蒸留した「働き方の前提知識・判断軸」。個人スコープ / チームスコープを持つ。
- **蒸留（distillation）**: 履歴 → ルールを抽出する処理。
- **適用（retrieval / apply）**: 作業前に該当ルールをAIの前提として読み込ませること。
- **昇格（promote）**: 個人ルールをチームルールに引き上げること（暗黙知→形式知）。
- **確度（confidence）**: ルールの確からしさ（high / med / low）。適用実績や矛盾で変動しうる。
