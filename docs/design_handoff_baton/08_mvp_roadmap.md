# 08 · MVP ロードマップ & 受け入れ基準

依頼者と合意した進め方: **「動くものを薄く一本通す（縦に貫く）→ 広げる」**。いきなり認証・チーム・定期ジョブから入らない。まず自分（依頼者）が毎日使えるものを作り、ドッグフーディングで磨いてからチームへ。

> **前提は §00 で確定済み:** プラットフォーム=GCP（Vertex AI Gemini / ローカルはMock）、ユースケース=「調査→Markdownレポート化」、スタック=§00 §0.2。以下フェーズ0の未確定事項は**すべて確定済み**（記録として残す）。

---

## 8.0 全体方針（鉄則）
- **最初の1ユースケースに絞る。** 第一候補「調べてレポートにまとめる」（§00 未確定 — 実装前に確定）。これが決まらないと成果物形式・execute プロンプトが決まらない。
- **認証・チームは後回し。** 個人ツールとして成立させる。ただしDBは最初から `workspace_id`/`owner_user_id` を持たせ、作り直さない（§02）。
- **蒸留は手動から。** 全自動の定期ジョブはフェーズ3。理由は §06.4。

---

## フェーズ0: コア体験の核を決める（✅ 確定済み — §00）
実装前の合意形成は完了。以下は確定内容（記録）。
- 最初に通すユースケース1つ → **「競合/市場の調査 → Markdownレポート化」で確定**（§00 #1）。
- そのユースケースでの「AIの実作業」の具体（入力・成果物・完了の定義）→ **§00 §0.3 に記載**。
- 成果物の置き場・形式 → **専用 `artifacts` テーブルに Markdown、バージョン管理あり**（§00 #2 / §02.6）。
- **成果物（このフェーズ）:** 入出力仕様＝§00 §0.3。

---

## フェーズ1: 技術の土台（個人で動く最小構成）
- スタック確定 → **§00 §0.2 で確定**（React+TS+Vite / FastAPI on Cloud Run / Cloud SQL(Postgres) / Cloud Tasks / Vertex AI Gemini。ローカルは Mock）。
- データモデル実装（§02）。単一 workspace・単一 user の固定値で開始。
- 看板ボードUI（§03）: 5レーン、カード、DnD、ドロワー（detail）、コメント。
- ステータス・ステートマシン（§05.6）とレーン移動。
- **AIにまかせる（execute）を1ユースケースで実装**（§07.2/7.3, ジョブ＋進捗 push）。
- **受け入れ基準:** シードのタスクを「AIにまかせる」→ 進行中で進捗が出て → レビューに移り成果物とハンドオフコメントが付く。人がコメントで介入できる。

---

## フェーズ2: 個人ツールとして実用化 ＋ ナレッジ最小ループ
- 壁打ち→分解（§01.6, §07.4）: chat モード、`propose_subtasks`、`confirmBreakdown` で子カード生成＋先頭AI子の自動着手。
- **ナレッジ機構（主戦場）:**
  - 適用（retrieval）: 着手前にルールを注入＋UIで可視化（§06.3, §03.3.2b）。
  - 手動蒸留: 「✧ 学ぶ」→ `propose_rules` → 人が採用/却下（§06.4a, §07.5）。
  - ナレッジ一覧オーバーレイ（§03.4）。個人ルールの表示・確度・適用回数・出典。
- ボードの永続化、AI実行のキュー安定化、失敗時リトライ＋通知（§07.2）。
- **受け入れ基準（§06.8 と同一）:** 完了タスクから学ぶ→採用→次の該当タスクの着手時に自動適用され、AIが「〜を前提に着手します」と宣言。同じ指示を繰り返さなくても2回目のアウトプットに反映される。
- ここで依頼者が**毎日ドッグフーディング**し、コンセプトの正否を体感する。

---

## フェーズ3: チーム展開（ここで初めて認証・権限）
- 認証（Google/SSO）、ワークスペースとメンバー、人担当が複数に。
- scope=team の実挙動を有効化（§06.5）。個人→チーム昇格が全メンバーに効く。
- 半自動→自動蒸留 ＋ 矛盾検出・降格・棚卸し（§06.4b/c, §06.6）。定期ジョブ（GCPなら Cloud Scheduler → Cloud Tasks/Run）はここで導入。
- AIの権限・コスト管理・監査ログ（§06.7, §07.6）。
- **受け入れ基準:** チームメンバー間でチームルールが共有・適用され、暗黙知が形式知として蓄積・改善されていく。

---

## 非ゴール（MVPで作らないもの）
- 定期ジョブによる全自動蒸留（フェーズ3）。
- ベクタ/意味検索（MVPはタグ一致。§06.3）。
- チーム/権限/課金（フェーズ3）。
- モバイル最適化（デスクトップ優先。§03.5）。
- 複数ボード/カスタムレーン（まずは固定5レーン）。
- リッチな成果物エディタ（ユースケース確定後に必要なら）。

---

## リスクと対策（要注意）
| リスク | 対策 |
|---|---|
| 蒸留が誤学習し、AIが変な癖を覚える | 手動採用を必須に。人の採用/却下ログを蓄積。自動化はフェーズ3＋矛盾検出/降格込み（§06.4/6.6） |
| AI実作業が長時間/失敗 | ジョブ化・リトライ・タイムアウト・人へのハンドオフ（§07.2） |
| ルールがブラックボックス化 | 着手時に適用ルールを必ず可視化、出典・適用回数を表示（§06.3, §03） |
| ユースケース未確定のまま実装 | フェーズ0を必ず先に。§00 のオープン論点を依頼者と潰す |
| チーム機能を先取りして複雑化 | 個人で成立してからフェーズ3。DBだけ拡張余地を持たせる |
| APIコスト | タスク単位のトークン/コスト記録、件数・履歴の上限管理（§07.6） |

---

## まとめ（実装者へ）
Grow の価値は「看板」ではなく **「履歴が資産になり、AIが使うほど自分にフィットする」** ループにある。UIは §03/04 に忠実に再現しつつ、**§06（ナレッジ機構）と §07（AI連携）が製品の心臓** であることを忘れないこと。まずフェーズ0でユースケースを1つ確定し、フェーズ1→2で「学ぶ→適用」の最小ループを個人で1周させる。そこが回れば、この製品は成立する。
